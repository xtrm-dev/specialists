§1The concept

A chain template is a named, reusable shape for a multi-step specialist chain. It encodes three things: the step DAG (which roles run, in what order), the per-step contracts (what each step promises and demands), and the overlay obligations (which gates apply, and at what severity). It exists because ad-hoc dispatch produced inconsistent chains — across ninety-six session reports from three repositories, the same shapes recurred. Naming them turned "what chain do I run" from per-bead reasoning into per-bead selection.

The system separates selection (which template fits this bead, decided by the bd create hook and sp chain plan) from composition (pouring the formula, wiring the dependency edges, reviewing the resolved shape) from execution (running the steps, firing the gates). This document concerns itself with what templates exist and what overlays apply — it is the philosophy, not the executable.

One source of truth

When this document and a .formula.json disagree on step shape, the formula wins (it is executable). When this document and using-specialists-v4 SKILL disagree on intent, this document wins (the SKILL is derived from this). When this document and substrate.md §6.9.10 disagree, substrate wins post-landing; this wins pre-landing. The pre- vs post-substrate split is deliberate — substrate is the future canonical, this is the bridge canonical.

§2The catalog at a glance

Thirteen default templates ship in docs/design/roadmap/chain-templates/. Each uses only package-tier specialists (cross-repo defaults). Per-repo extension via extends is supported (see the market-data quant-validation example in chain-templates/README.md).

The compact view below names each template and its primary use. Full per-template detail — mermaid step diagrams, variables, overlay defaults, anti-patterns — lives in the companion chain-templates.md §2.

The catalog is evidence-backed — every template name maps to a recurring shape observed in real session reports. New templates are added through the evolution protocol (§6), not on speculation.

§3Overlays — the gates that compose

An overlay is a set of gates that applies across templates rather than being shape-specific. Overlays are how the catalog evolves without re-defining each formula every time we learn something new. There are three today.

§3.1 Iron — review hardening

The Iron overlay adopts six concepts from Jane Street's Iron code review system. We do not import Iron code; we restructure prompts, add one cheap specialist (obligations-scanner), and let scrutiny decide depth.

Iron conceptOur adoption
Scrutiny profilesSCRUTINY: low|medium|high|critical on every bead
Mandatory seconderscode-sanity as gate (not advisor) on production diffs
Obligations trackingobligations-scanner blocks unstructured TODO/FIXME/HACK; TODO(<bead-id>) permitted
ddiff re-reviewOn PARTIAL, reviewer audits only the delta; prior approvals carry forward
Formal release gatesReviewer outputs a Release Checklist every chain
Auto-escalationReviewer raises floor on auth · config/specialists · lockfiles · migrations · permissions/hooks

The full auto-escalation table, the illustrative reviewer-prompt checklist lines, and the manual-git-workflow rationale are in chain-templates.md §3.1.

§3.2 QA — the gap between code and review

Iron hardens the review pipeline but leaves a precise gap: who turns a production diff into the right behavioral tests, smoke checks, and telemetry assertions before the reviewer sees it? The answer is not the executor (busy implementing) and not test-runner (only executes). The QA overlay introduces test-engineer and upgrades test-runner.

The full role table, output schema, failure-routing matrix, and channel-message kinds (when channels v0 ships) are in chain-templates.md §3.2. Implementation is tracked under epic unitAI-sfwe1; chain_template formula integration under unitAI-f9kku.

§4Severity decides depth

SCRUTINY is a required field on every bead's change-contract. It drives overlay obligations across the system — same template, different scrutiny, different active gate set.

TierCost-of-being-wrongOverlay floor
lowTrivial · easily revertedIron minimal · QA optional · DevOps N/A
mediumNormal · production codeIron mandatory · QA recommended · DevOps opt-in
highSignificant · cross-cuttingIron mandatory · QA mandatory · security-auditor on sensitive surfaces
criticalSevere · auth, money, irreversibleAll above · security-auditor as gate · behavioral QA evidence in checklist

Two rules govern movement between tiers. Auto-escalation raises the floor when scope hits a sensitive surface — the reviewer never lowers a declared scrutiny but can raise it (a bead declaring medium on src/auth/ is treated as critical). Operator override can declare any bead critical regardless of the apparent size — cost-of-being-wrong is operator judgment, never tool judgment.

The per-template default scrutiny floors and the full auto-escalation table are in chain-templates.md §4.

§5Composition — Layer-1 plus overlays

Composition is deterministic — same inputs always produce the same chain. The order matters:

  1. Layer-1 template shape from the .formula.json. This is the foundation; overlays cannot remove these steps.
  2. Overlay obligations applied in declaration order: Iron → QA → DevOps. Each overlay inserts steps at well-defined positions (Iron at the tail; QA between writer-role and code-sanity; DevOps between test-runner and code-sanity when applicable).
  3. Auto-escalation pass. The reviewer's auto-escalation table runs against the bead's scope and raises the effective SCRUTINY floor. Floor changes can promote recommended-or-conditional cells to mandatory (never demote mandatory).
  4. Operator inserts. sp chain insert <chain-id> <role> --before|--after <step> applies operator-requested additions. Cannot remove Layer-1 or overlay-mandatory steps.
  5. Coordinator entry-gate (post-substrate). The chain coordinator (substrate §4.3 role 1, §6.3.1) re-validates the composed chain from fresh context and may propose additional inserts within its autonomy policy. Pre-substrate: this step does not exist; the dispatcher hands directly to step-1.

Overlays do not conflict by design — they target different gate positions. If two overlays propose the same step (e.g., both QA and DevOps want test-runner), the later overlay extends the earlier's contract rather than duplicating the step.

The per-template overlay matrix — the big table where rows are templates and columns are overlay obligations — is in chain-templates.md §5. Composition mutual-exclusion and extension rules are in §6 there.

§6Evolution — folding lessons back

The catalog and the overlays are living artifacts. The evolution protocol decides when and how to add a new template, a new overlay, or an updated per-template obligation.

When to add a new template

Three or more distinct chains across two or more repositories exhibiting the same Layer-1 shape that none of the existing templates covers cleanly. The pattern must be repeated (not a one-off), stable (not in flux), and distinct (not a degraded form of an existing template). Evidence comes from .xtrm/reports/*.md across repos.

When to add a new overlay

A gate or set of gates applies broadly across multiple templates (not template-specific shape) and represents a concern orthogonal to existing overlays. Iron, QA, DevOps are the three so far — review hardening, behavioral validation, operational validation. Each is a distinct concern, each applies across multiple production-diff templates.

When to update a per-template obligation

When the matrix changes — e.g., code-quick raising code-sanity from optional to recommended because we observed too many post-merge regressions on trivial-looking changes — the change lands in four places: the §5 matrix (canonical declaration), the reviewer prompt (enforcement), the bd create hint (operator visibility at bead-create time), and the CHANGELOG note for the next release.

When to fold session findings

Per-session: the orchestrator captures observations as bd remember memories tagged convention or behavioral. Per-design-pass: a design session like this one reviews recent memories and folds the durable lessons here, into the reviewer prompt, and into the matrix. The revision history lists the date of each substantive fold so the lineage is greppable.

Substrate alignment

Substrate.md §6.9.10 lists the same template roster from a substrate primitive perspective — each template becomes a row in substrate.chain_templates; each overlay becomes a declarative rule in substrate.mandatory_layers. When substrate lands, the migration is mechanical (the data is already shaped this way). This document remains the philosophy that explains why the data is shaped the way it is. Substrate will execute; this document teaches.
