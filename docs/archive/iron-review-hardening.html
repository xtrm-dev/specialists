Iron-Inspired Review Pipeline Hardening

Adopt Jane Street's Iron code review concepts — scrutiny profiles, ddiff re-reviews, mandatory seconders, obligations tracking, formal release gates — into the specialists pipeline. Six tightly-scoped child chains delivered in three waves. Pipeline becomes risk-aware instead of one-size-fits-all.

Epic iron-review-hardening Priority P1 Targets reviewer · code-sanity · executor · debugger · NEW obligations-scanner · skill + CLAUDE.md Execution manual orchestrator (no dispatch)
Execution mode for this epic: manual orchestrator-direct edits. Explicit exception to skill rule #13 ("orchestrator NEVER edits code directly"). Rationale: the epic restructures the very specialists used for normal dispatch, so bootstrapping via those specialists is circular. All other epics continue to use the dispatch model.

Why this exists

Our current pipeline applies the same review depth to a 2-line test tweak and a payment-flow rewrite. code-sanity is advisory and frequently skipped. After a PARTIAL verdict, reviewers re-audit the entire diff instead of just the delta. In-code obligations (TODO/HACK/FIXME) land in production without a gate. Files like src/specialist/runner.ts or config/specialists/*.json inherit whatever scrutiny the bead author requested — not what the content demands.

Iron solved exactly this set of problems with a small set of orthogonal primitives. We adopt the concepts directly. We do not import any Iron code; we restructure prompts, add one cheap specialist, and rewrite the pipeline skill.

Old vs. New pipeline

What changed structurally: two boxes got a solid border (mandatory, not advisory). One box was added (obligations-scanner). The PARTIAL loop went from dashed-red (re-audit everything) to dashed-green (ddiff scope only). Reviewer became risk-aware and emits a machine-readable Release Checklist that future sp merge can enforce.

SCRUTINY taxonomy

A first-class bead-contract field. Reviewer reads it and tiers its behavior. Beads without the field default to medium (backwards compatible). Reviewer can auto-escalate the floor based on the diff surface — author's stated level is a floor, not a ceiling.

low
Seconder-only. code-sanity OK is sufficient. Reviewer checks contract compliance + spot-checks 2–3 files. No blast-radius requirement. → Test-only diffs · docs · isolated config tweaks.
medium
Default. Current reviewer behavior. code-sanity required. security-auditor required if sensitive surface. → Most production diffs.
high
File-by-file sign-off. Mandatory gitnexus_impact evidence. code-sanity + security-auditor both required. Ddiff re-review enforced. → Auth · payments · schema · migrations · agent core.
critical
High + second opinion. Mandatory obligations-scanner + at least one independent reviewer turn (or overthinker premortem). → Credential storage · model routing · specialist permission boundaries.

Git state precondition before chain dispatch

New skill rule. Before dispatching any specialist chain whose work depends on prior chain output, the orchestrator MUST verify git state is clean and the base branch contains the prior chain's commits. This is the most common cause of "the chain forked off the OLD baseline" debugger-restitch loops documented in the existing skill.

Specialist worktrees fork from the current HEAD of the orchestrator's branch at dispatch time. If prior chain edits aren't merged in yet, the new chain works on a stale base, will conflict at integration, and debugger-restitch becomes mandatory. The fix is upstream: don't dispatch until prior work has landed.

Required pre-dispatch checks

# 1. Working tree is clean (no uncommitted edits that would be inherited or lost)
git status                          # MUST report "working tree clean"

# 2. Latest commit on the dispatch branch contains prior chain work
git log -1 --oneline                # confirm prior chain's commit is HEAD or in HEAD's history
git log main..HEAD --oneline        # confirm prior chain's branch was merged in

# 3. No orphaned worktrees from prior chains
git worktree list                   # every chain worktree from prior dispatches should be removed
git worktree prune                  # drop stale metadata

# 4. If working on an integration branch:
git log integration/<date>..HEAD    # MUST be empty (orchestrator's branch is in sync)
Decision rule: if any of the four checks fail, finish the merge / commit / cleanup first. Do not dispatch. A specialist forked from a stale base produces conflict work that costs more turns than the time saved by dispatching early.

When this rule applies

ScenarioPrecondition strictness
Sequential chains in same epic where child.B depends on child.A's editsStrict. child.A must be merged before child.B dispatch.
Parallel chains in same epic with disjoint file scopesRelaxed. Each dispatches off the shared base; integration phase reconciles.
Chain after orchestrator-direct edit (e.g., this epic)Strict. Orchestrator commits + pushes their direct edits before dispatching any chain that references them.
Standalone chain (no upstream dependency)Relaxed. Just git status clean.

Worked example — high-scrutiny chain end-to-end

Channels in an Iron review workflow

The channel substrate changes how specialists coordinate during a review chain. Instead of the orchestrator passing outputs between jobs, each specialist subscribes to the shared channel and wakes on the messages it cares about. Prior verdicts from code-sanity and obligations-scanner stay in the channel history — the reviewer reads them directly, no orchestrator forwarding. On a PARTIAL verdict the executor wakes from the reviewer's verdict:PARTIAL message, fixes only the flagged delta, and posts a new turn; on re-review the reviewer reads the channel history to identify prior approvals and audits the delta only. Ddiff mode emerges from the channel naturally, not from orchestrator state.

Medium scrutiny — channel walkthrough

  1. Open. sp ch open executor code-sanity obligations-scanner reviewer --bead <id> --stop-on pass. Channel chain:<bead-id> opens; subscriptions registered: executor (verdict:me), code-sanity and obligations-scanner (turn:peer), reviewer (turn:peer + verdict:me×2).
  2. executor writes diff, posts kind=turn, summary="…", refs={result_id} to channel.
  3. code-sanity and obligations-scanner wake in parallel from the same turn:peer event — no orchestrator coordination. Each runs independently and posts its kind=verdict targeted at reviewer.
  4. reviewer wakes once both targeted verdicts have arrived. Reads full channel history (turn + both verdicts). Applies medium depth. Posts kind=verdict, verdict=PASS with Release Checklist.
  5. Stop condition stop-on:pass fires → channel closes → sp merge.
On PARTIAL (step 4): reviewer posts verdict:PARTIAL, target=executor, findings=[…]. Executor wakes (verdict:me), reads only its delta findings from the channel message. Fixes, posts a new turn. Reviewer wakes again; reads the channel history to identify what it already approved in its prior verdict; audits only the new diff range. Ddiff is the reviewer reading its own prior message — no orchestrator state needed.

High scrutiny — channel walkthrough with ddiff loop

  1. Open. sp ch open executor code-sanity security-auditor obligations-scanner reviewer --bead <id> --scrutiny high --stop-on pass. Reviewer's spec applies auto-escalation: src/auth/* → floor=high confirmed; lockfile change → security-auditor mandatory.
  2. executor writes diff (auth + migration), posts kind=turn with refs={diff_range} to channel.
  3. code-sanity, security-auditor, and obligations-scanner all wake in parallel from the same turn:peer subscription. No orchestrator coordination — the channel delivers the event to all three simultaneously.
  4. Each posts its kind=verdict targeted at reviewer: verdict:OK, verdict:sec={findings}, verdict:CLEAN.
  5. reviewer wakes once all three targeted verdicts have arrived. Applies high mode: file-by-file sign-off, gitnexus_impact mandatory. Posts verdict:PARTIAL, target=executor with per-file findings. The partial verdict body records which files are already approved — this becomes the ddiff baseline.
  6. executor wakes (verdict:me), reads the partial findings from the channel message — sees exactly which files need work. Fixes those files only. Posts kind=turn with refs={new_diff_range} pointing at the fix commits.
  7. reviewer wakes in ddiff mode: reads the channel history to find its own prior verdict:PARTIAL, reconstructs the approved-file list, then audits only the new diff range from step 6. Prior-approved files carry forward.
  8. Posts verdict:PASS + full Release Checklist → stop condition fires → channel closes → sp merge.
What the channel buys at high scrutiny: the reviewer's step-5 partial verdict is a first-class message in the channel. When it wakes in step 7, it reads its own prior message to reconstruct "what I already approved" — no orchestrator state, no external memory, no re-audit of unchanged code. The three parallel seconders also coordinate without the orchestrator: they all wake from the same turn event and each posts independently to the channel. The reviewer wakes only once all three have posted their targeted verdicts.
