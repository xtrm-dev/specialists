Two further conversations after the report was written produced two new epics with 4 children each. Captured here so the handoff is complete.

### KPI investigation findings

Inspected `specialist_job_metrics` table (rich schema: job_id, specialist, model, status, elapsed_ms, total_turns, total_tools, tool_call_counts_json, token_trajectory_json, context_trajectory_json, stall_gaps_json, run_complete_json) and discovered:

- **Writer exists** (`aggregateJobMetrics` in `src/specialist/observability-sqlite.ts:1905`) but **only called on-demand** via the hidden `sp db extract` subcommand. Supervisor's terminal-status path doesn't call it; result: empty table under normal operation.
- `sp db extract` and `sp db stats` are hidden — not in `sp db --help`.
- **Ad-hoc state change**: ran `sp db extract --all-missing` during the investigation; **960 historical rows** now populated. Confirmed real signal — outliers visible (e.g. `executor + gpt-5.4-mini` 18 runs, ~4050s avg elapsed). Important: that elapsed-time outlier is almost certainly accumulated waiting-state from forgotten keep-alives, not active model time. Active runtime split is needed.

Key design choices captured in the new epic:
- Three runtime metrics required (`total_runtime_ms` / `active_runtime_ms` / `waiting_ms`) computed from existing `specialist_events` state transitions.
- Per-spawn payload measurement must include **everything the specialist spawns with**: system_prompt, task_template, mandatory_rule (one entry per rule), skill (one entry per skills.paths), pre_script_output, memory (static/dynamic/gitnexus), bead_context (own/parent/blocker:N).
- Both **tokens (approximate cl100k_base-style)** AND **bytes (UTF-8 length)** — bytes are universal across tokenizers.
- Surface in `sp feed` / `sp ps` / `sp result` / `sp run --json` / `sp db stats`.
- Bundle into KPI table as `startup_payload_json`.

### Reconciliation finding

CLAUDE.md drift across `~/dev/CLAUDE.md`, `~/dev/specialists/CLAUDE.md`, `~/dev/xtrm-tools/CLAUDE.md` is unmanaged. Tool-injected primes (bd, gitnexus, sp) evolve faster than the static docs. Concrete gotchas observed *this session* that agents had to rediscover:

- `bd create --parent <epic-id>` (auto-`.N` children) was missing from CLAUDE.md bd reference; user had to surface it before the KPI epic was filed correctly.
- `bd kv set "memory-acked:<id>"` (memory gate) — caught me mid-session when probe bead closed without ack.
- `sp merge` is broken when `status.json` is cleaned — not documented as a known limitation; surprised me when forced to manual fallback.
- `bd query` for SQL-like filters, `bd dep relate`, `bd dep <a> --blocks <b>` shorthand — all documented but not loud enough.

Proposed mechanism: sentinel-marker-managed fragments (mirror `sp init --sync-defaults` pattern) + `xt claude-sync` script + `xt doctor` drift check + tool-injected hot-tips at session start.
