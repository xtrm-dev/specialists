# Session Report — 2026-05-07

## Summary

The session converted the “should specialists and xtrm-tools become a monorepo?” question into a concrete split-repo hardening plan: keep the repos/packages separate for now, but make them contract-dependent through CI/CD, npm-pack validation, fresh-machine install smokes, and explicit ownership of Category A specialists runtime assets versus Category B xtrm-managed filesystem payload. Twelve successful overthinker audits, plus three failed/stalled first attempts in xtrm-tools, checked specialists `config/`, xtrm-tools `.xtrm/`, npm payloads, release workflows, install order, package naming, and top-level `AGENTS.md`/`CLAUDE.md` context. The major decisions are: do not release xtrm-tools yet; remove the `xtrm-tools@^2.0.2` self-dependency; likely migrate the canonical xtrm package to scoped `@jaggerxtrm/xtrm` with bins `xt`/`xtrm`; do not make specialists hard-depend on xtrm as a normal npm dependency; require xtrm-first bootstrap (`xt install/init` before `sp init`); and add fresh-machine smokes that prove both packages work without local repo clones, cross-repo symlinks, or `SPECIALISTS_REPO_PATH`. No source code or skill files were modified; the output was a detailed issue board and this handoff report.
